## Current State

The repo currently has only the domain bundle and task/plan artifacts. There is
no checked-in UI application tree yet, so this plan starts from the supplied
Stitch screen and the UI domain page rather than from existing component code.

The UI contract already defines the operator audiences, the top-level surfaces,
and the approval/navigation constraints. The supplied HTML scaffold adds the
exact visual grammar to preserve: a compact sticky top bar, indigo primary
accent, light surfaces, JetBrains Mono for IDs, and a dense content canvas.

## Objective

Create the UI shell as a production-intent web application entry point that
matches the supplied Stitch screen and token set exactly enough that future
pages can be layered on top without redesigning the shell. The result should be
role-aware, project-aware, and visually stable across the operator views defined
in `docs/domain/ui.md`.

## Out of Scope

- Do NOT invent a new visual language or new primary color palette.
- Do NOT implement full backend workflows for approvals or reconciliation in
  this task.
- Do NOT broaden the UI into unrelated admin, mapping, or migration screens.
- Do NOT change the domain docs except where a UI wording clarification becomes
  necessary for the shell itself.
